MESH WALL TOOL (USED IN HOGWARTS LEGACY)

A seemengli normal spline mesh tool, with the unique feature of only using Spline Mesh Components where they are actually necesary, everywhere else it will try to use Instanced Static Mesh Components, unless there is only 1 instance, in which case it will use a regular Static Mesh for that section.

It has other features, should as letting you pick a specific Star and/or End mesh, to add pillars in between each section. Or pillars as start/end meshes.

A video

(Open in new tab to see properly) In the details panel you can specify a list of meshes to pick from, randomly, but can also assign weights to alter their probability.

A Stone Wall that needs to curve.

Only the sections that actually have enough curvature will use Spline Mesh, all the other are Instanced Static mesh.

This Fence wall does not need to display any curved sections because of its design.

So nothing uses Spline Meshes in this case.

SIMPLIFIED COLLISION TOOL (USED IN HOGWARTS LEGACY)

A Spline-Based simplified collision tool. I originally created a simpler version of it for my game Bandits, but suggested to the Hogwarts Legacy guys they needed it something like it. So I created this advanced version for that game.

The tool is a single InstancedStaticMeshComponent with a clean box mesh, scaled to fit each segment.

The tool is quite cheap to process, and even though it shouldn't really need to be too long, it can and it barely hitches the editor.

(Open in new tab to see properly) The tool is C++ based, so I could expose the InstancedStaticMeshComponent Collision properties directly to the Actor details panel.

The wall acounts for the spline courve to only generate new instances/sections where the delta rotation has exceded the max allowed delta (configurable parameter).

Here you can see clearly each section that was created in this case.

The wall can also be configured to handle pitch rotation, in case you needed to add colission on a cavern or stairs.

One super neat feature of the wall is how the instances/segments always allign perfeclty on the outside corner, so there is no gap or space to get snaged by. Prised be the math.

HOGWARTS LEGACY - PROXY TOOL

In this game the level were built in many sublevels used as layers, which made HLODs not a viable option. So I created a system that would use a SetupMap to load all the levels used on a buildings and capture a snapshot of it to generate a Proxy from, that included everything. This also suported the Winter variation that was covered in snow. At the end ProxyContainerActor was played on the LOD level automatically. We created a TeamCity job for this, that would run constantly, so any work artist did on any level would be shortly reflected on the LOD Proxies.

Each building we wanted a proxy of, needed to have a setup on one of the setup maps. There you would place a volume and configure it through a handy Utility Widget.

The result was a proxy of all the sections that had impostors such as windows, vines, and small detail. A regular Proxy, and a Winter version. In the end these would all be assembled back in the ProxyContainerActor that gets placed in the LOD map.

Here is the proxy with only the flat stuff, which has an extremely low poly count.

This is the proxy, that does not bake any of the meshes that had impostors in them. This saved an tremendous amount of triangles, and would actually help with the silouete. At the end both meshes are assembled in the ProxyContainerActor and looks like a normal proxy with all its elements.

The system also extended to the Landscape LODs, which used the same ProxyTool, but unbound, to generate a proxy of all the actors and meshes on the tile. There it performed specific steps for big rocks, hills, long spline tools, roads, and so on.
Everything combined was quite literally everything outside of the player nearby readius, including the entire Hogsmeade Town and the Hogwarts Castle, which was quite a lot of work to get right!

HOGWARTS LEGACY - FAR FOLIAGE

A UE4 World Composition optimization that handles the far away foliage without ugly Tile LOD proxies. Also good to hide level streaming pops, as this system functions independently of it. As you move around the system updated which far foliage instances are rendered.

1st step was to capture all the big-enough overworld foliage and store them in a pre-computed list. We created a TeamCity job that would do this step daily, so any work artists did would be reflected on the farfoliage not long after.

At runtime there is a radius around the player, here represented by a big sphere, and outside of it normal Foliage is no longer rendered, as the foliage max-draw distance was set to match the radius of the far foliage system.

Here I've selected the normal foliage, and everything beyond the sphere is Far Foliage, which is a collection of Impostors.

As you move around, the run-time system evaluates which far foliage to render, while the regular foliage just uses the native foliage optimization of Unreal and fades out foliage after a specified max-draw distance.
Aditionally the system has other optimizations, such as only evaluating after you've moved enough, that the tiles are inside the view frustrum, and that they are close enough to be updated.

RPG SIMPLE DIALOGUE SYSTEM AND SETUP

The dialogue system has all you need without reinventing the wheel. It just uses Blueprints, and 4 custom C++ Nodes to handle everything. It supports regular lines, questions, branching dialogue, a fallback concluded state, interaction with the world, etc.

The dialogue system implemented in game. At the end the player asks the NPC to unlock a door for them.

The first pair of nodes are Latent Blueprint nodes, they start and won't Complete until they get their callback. The second pair are so you can print the lines however way you want. And the last two are the callbacks to let the Latent Nodes know they can now Complete.

Inheriting from a Parent class that handles all the nuance, child blueprints only need to worry about their scripted dialogue.

The actor/script is attatched to the NPC or object that will be hosting it.

En example of how the blueprint dialogue script looks like. Except for the custom Request line/question nodes, it is just Blueprint.

GRID MATERIAL

Very useful for prototypes or low poly games. Its a material that lets you use a Material Instance to pick the color of each one of its 16 sections.

The Matereial splits the UV into 16 sections and asignes a color to each one. It would be extended to also include other things if the project needed it.

On Blender I've set up the UV to match the colors sections into the UV grid.

This way all the prototype assets use a single material with a single draw call that lets you have up to 16 colors per asset.

Then on the Material Instance for these assets you select the colors each grid section should have. Super simple.

ASTERMINER (UE5, WIP)

Asterminer is a game where you control a truck-ship to explore around, collect crystals, and load them into a train.
It is based on an earlier game I made for Android/iOS, which had over 50k+ premium downloads.

This is a prototype of a 6 wheeled rover with independent suspension on each wheel. The vehicle is super smooth and satisfying to drive. It is all interpolation with no physics.
